## The real problem (model + prompt, not refactor)

The `ministral-3:3b` captain LLM defaults to ping every turn, regardless of:

- **Ship class** тАФ DC ships never dropped DCs.
- **Fleet directive** тАФ the fleet commander repeatedly issued "engage
  aggressively" with explicit destinations; captains ignored.
- **Contact confidence** тАФ contacts at 1.0 confidence got pinged, not
  engaged.
- **Damage state** тАФ `red-dd-01` kept firing torpedoes after taking three
  bow hits in compartment 0 (presumed torpedo-room damage).

The 4% non-ping decisions are essentially noise. **The captain layer is not
making decisions тАФ it is stalling.** Pinging is the LLM's "I do not know
what to do" output.

### Three observations that point at fixes

1. **Captains never see geometry as numbers** тАФ only as text in a 1800-token
   JSON blob. A 3B model cannot reliably translate "fleet destination is
   `[985.4, 108.7]`, your position is `[630, -350]`" into "set heading 119,
   speed 10."
2. **Captains never see role-specific doctrine** тАФ every captain gets the
   same prompt, then must *infer* "I am a depth-charge destroyer, I should
   drop depth charges." That inference does not happen at this model size.
3. **Capability gating is silently masking failed decisions** тАФ when
   `red-dd-01` kept trying to fire after damage, those rejected attempts
   vanished into "run with no apply." The captain LLM has no signal that
   the action was rejected, so it keeps proposing the same broken action.

## Recommended next move

Wire `tactical.doctrine_for` and `bearing_to/range_to` into
`_build_ship_summary` as a `SENSORS_BRIEFING` block in the captain prompt:

```
SENSORS_BRIEFING:
  fleet_destination_bearing: 119.4┬░
  fleet_destination_range_m: 1023
  best_contact: bearing 280.5┬░, range 4200 m, confidence 0.85, in_torpedo_envelope=true
  doctrine_recommendation: ENGAGE_TORPEDO
  suggested_action: fire_torpedo bearing=280.5 run_depth=120 enable_range=1500
```

Plus one paragraph in `ai/ship_commander_system.md`:

> *The `SENSORS_BRIEFING` contains pre-computed tactical answers. Default
> to following `suggested_action`. If you deviate, prefix the summary with
> `deviate:` and state the reason. Do not compute geometry yourself.*

Predicted outcome on a re-run with the same model and missions:

- Ping ratio drops from 96.5% to under 50%.
- Depth-charge ships start dropping depth charges.
- Captains transit toward fleet destinations.

This is contained: `ai_orchestrator.py` + one prompt file. No `loop.py`,
`core.py`, or `control/` changes.

## Outstanding follow-ups

- **`ai.tool.apply` payload regression**: my Phase 2 code emits
  `{"ship_id": ..., "tool": "ActivePingAction"}` instead of the prior
  `{"ship_id": ..., "tool": "active_ping", "arguments": {...}}`. No UI reads
  this event today, so it is not blocking, but it should be enriched to
  include the action's parameters via `dataclasses.asdict()`.
- **Failed-action signal back to LLM**: when `ShipControls` returns
  `ControlResult.fail(...)`, the orchestrator does not see the error. On
  the next prompt the LLM sees no record of having tried, encouraging the
  same loop. Consider surfacing the last failure as `last_action_failed:
  {tool, error}` in the ship summary.
- **Sound destroyers don't coordinate EMCON** тАФ every ship pings constantly,
  giving away their positions. Game design / fleet-prompt issue.
